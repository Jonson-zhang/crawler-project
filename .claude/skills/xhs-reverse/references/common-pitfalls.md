# 常见坑

## 坑 1：code=0 items=0 — 签名或 Cookie 不完整

**现象**：homefeed 返回 `success=True, code=0`，但 `items=[]`。

**容易误判为**："需要 web_session（登录态）才能拿数据"。

**真实原因**：Cookie 引导阶段的某个请求没带完整签名头，导致服务端下发了降级的 cookie。这些 cookie 足够让接口返回 HTTP 200，但不足以拿到数据。

**排查方法**：
1. 用真实浏览器的 cookie 发同一个请求 → 有数据 = 你的 cookie 有问题
2. 检查自己生成的 x-s 解码后的 x3 前缀是否为 `mns0301_`（不是 mns0201）
3. 检查请求头是否缺了 x-s / x-s-common / x-t / x-b3 / x-xray 中的任何一项
4. 回查引导流程中哪个步骤的签名不完整

**教训**：访客 Cookie 可以拿数据，不要因为 code=0 就放弃。

## 坑 2：引导阶段的请求没签名

**现象**：shield/webprofile 和 activate 返回 200，但引导后的 cookie 不能用。

**容易误判为**："这些请求不需要签名"、"接口本身就支持无签名访问"。

**真实原因**：这些接口确实可以不签名就返回 200（也是降级模式），但签名完整的请求能拿到"完整权限"的 cookie。

**排查方法**：用浏览器抓取一份带完整签名的 shield/webprofile 请求，和自己的请求做逐行对比。关键差异通常在 x-s-common 的字段数和长度。

**教训**：引导阶段和业务阶段的每个请求都要带完整签名头。

## 坑 3：硬编码版 x-s-common 在部分接口失效

**现象**：简化版 x-s-common 在 homefeed 通过，但 shield/webprofile 被拒。

**容易误判为**："x-s-common 没问题，已经验证过了"。

**真实原因**：不同接口对 x-s-common 的校验严格程度不同。homefeed 放行的简化版，shield/webprofile 可能不放行。

**排查方法**：对每个端点单独验证，不要把一个端点的结果推广到所有端点。

**教训**：每个请求独立验证。

## 坑 4：补环境跑通 ≠ 结果正确

**现象**：sign.js 不报错，能输出字符串。

**容易误判为**："环境补好了，可以用了"。

**真实原因**：环境变量缺失时 VMP 可能会走 fallback 路径，产出低版本结果（mns0201 而非 mns0301），但不会报错。

**排查方法**：解码 x-s，检查内部 x3 字段前缀。必须是 `mns0301_` 才算正确。

**教训**：补环境不是为了跑通，是为了跑对。必须和浏览器输出一致。

## 坑 5：补环境没有用原型链

**现象**：Object.assign 平铺的属性能跑通简单页面，但 VMP 字节码执行时报 `apply on undefined`、`is not a constructor`。

**容易误判为**："属性值不对，需要换个值"。

**真实原因**：VMP 解释器会做深度类型检测：`typeof`、`instanceof`、`toString()`、原型链遍历。

**排查方法**：给每个浏览器对象正确的原型链，所有函数标记 native toString。详见 [env-patching.md](env-patching.md)。

**教训**：原型链补环境是必须的，不是可选的。

## 坑 6：abRequestId / sec_poison_id 缺失

**现象**：其他 cookie 都在，就少一两个，接口还能返回 200。

**容易误判为**："少了也不影响"。

**真实原因**：某些接口可能不校验这两个值，但完整引导链路中后续的请求可能会校验。少了就静默降级。

**排查方法**：对比真实浏览器的 cookie 字段数。引导完成后逐一核对。

**教训**：Cookie 一个不能少，列表逐项核对。

## 坑 7：webBuild 硬编码过时

**现象**：x-s-common 生成的 x4 字段使用了旧版本号。

**容易误判为**："webBuild 是固定值"。

**真实原因**：小红书每次发版 webBuild 会更新，旧的会被风控标记。

**排查方法**：每次逆向时从浏览器 Cookie 的 `webBuild` 字段获取当前值。

**教训**：不硬编码会随版本变动的值。
