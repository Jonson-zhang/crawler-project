# 常见坑

## 坑 1：code=0 items=0 — Cookie 不完整

**现象**：homefeed 返回 `success=True, code=0`，但 `items=[]`。

**容易误判为**："游客账号没有推荐数据"、"接口就是这样"。

**真实原因**：Cookie 引导阶段的某个请求没带完整签名，导致服务端下发了降级的 cookie。这些 cookie 足够让接口返回 HTTP 200，但不足以拿到数据。

**排查方法**：
1. 用真实浏览器的 cookie 发同一个请求 → 有数据 = 你的 cookie 有问题
2. 逐个去掉你的 cookie 中的字段，加到浏览器 cookie 中 → 找到缺失的关键字段
3. 回查引导流程中哪个步骤的签名不完整

**教训**：不要因为 code=0 就认为一切正常。cookie 不完整是静默失败。

## 坑 2：引导阶段的请求没签名

**现象**：shield/webprofile 和 activate 返回 200，但引导后的 cookie 不能用。

**容易误判为**："这些请求不需要签名"、"接口本身就支持无签名访问"。

**真实原因**：这些接口确实可以不签名就返回 200（也是降级模式），但签名完整的请求能拿到"完整权限"的 cookie。

**排查方法**：对比 RedCrack/xhshow 生成的 shield/webprofile 请求头和自己的请求头。关键差异通常在 x-s-common 的字段数和长度。

**教训**：引导阶段和业务阶段的每个请求都要带完整签名头。

## 坑 3：硬编码版 x-s-common 在部分接口失效

**现象**：5 字段硬编码版 x-s-common 在 homefeed 通过，但 shield/webprofile 被拒。

**容易误判为**："x-s-common 没问题，已经验证过了"。

**真实原因**：不同接口对 x-s-common 的校验严格程度不同。homefeed 放行的简化版，shield/webprofile 可能不放行。

**排查方法**：对每个端点单独验证，不要把一个端点的结果推广到所有端点。

**教训**：每个请求独立验证。

## 坑 4：eval 没找全 — mns 版本不对

**现象**：sign.js 能跑通，输出的 x-s 格式也对，但接口返回 406 或空数据。

**容易误判为**："环境没补对"、"参数传错了"。

**真实原因**：只找到 2 个 eval，少了 ds_v2.js。window.mnsv2 停留在 mns0201，数据接口需要 mns0301。

**排查方法**：解码 x-s，看内部 x3 字段前缀是 mns0201 还是 mns0301。

**教训**：搜 eval 不能只搜一次。反复搜、反复断点，直到确认没有遗漏。

## 坑 5：补环境跑通 ≠ 结果正确

**现象**：sign.js 不报错，能输出字符串。

**容易误判为**："环境补好了，可以用了"。

**真实原因**：环境变量缺失时 VM 可能会走 fallback 路径，产出不同版本的结果（如 mns0201 而非 mns0301），但不会报错。

**排查方法**：和浏览器的输出做逐字节对比，不能只看"有没有输出"。

**教训**：补环境不是为了跑通，是为了跑对。

## 坑 6：abRequestId / sec_poison_id 缺失

**现象**：其他 cookie 都在，就少一两个，接口还能返回 200。

**容易误判为**："少了也不影响"。

**真实原因**：某些接口可能不校验这两个值，但完整引导链路中后续的请求可能会校验。少了就静默降级。

**排查方法**：对比真实浏览器的 cookie 字段数。引导完成后应该是 11 项。

**教训**：Cookie 一个不能少，列表逐项核对。
