## 一、目的

网站 https://www.mihuashi.com（米画师）的 API 接口在请求头中要求 M-S 签名参数，
需要逆向该签名算法，实现 Python 爬虫获取漫画素材数据。

## 二、加密参数观察

1. 数据接口：`GET /api/v1/stalls?category=16&only_fast=false&order=2&page=1&per=20&state=forsale`
2. 请求头中的签名参数：
   - `M-S`：动态加密签名
   - `M-T`：Unix 秒级时间戳
   - `Web-Version: frontend`：固定值
3. 签名仅对 URL 路径（如 `/api/v1/stalls`）签名，不包含查询参数
4. 分类列表接口 `/api/v1/configure/art_categories/stall` 无需签名
5. 响应是明文 JSON

## 三、签名定位

重点需要逆向请求头中的 M-S 参数，在浏览器控制台中全局搜索 M-S，找到两处赋值位置，
经过测试真正起作用的只有一处：在 `http.B6ZEV3BE.js` 中：


其中：
- `u`：API 路径，如 `/api/v1/stalls`
- `t`：时间戳
- `n`：签名函数，来自 `http.NcR9CHD7.js`

## 五、补环境关键发现

### 5.1 WASM 初始化时的 DOM 读取（关键！）

在真实浏览器中对 `signtool_new()` 注入追踪代码，记录每一个 wbg 函数调用的顺序和参数：

```
=== signtool_new() 调用序列 ===
  __wbg_static_SELF                          → 获取 self 全局对象
  __wbindgen_object_clone_ref                → 引用 self
  __wbindgen_is_undefined = false
  __wbindgen_object_clone_ref                → 引用 true
  __wbg_crypto()                             → 获取 self.crypto
  __wbindgen_is_object = true
  __wbg_newwithlength(256)                   → 分配 256 字节缓冲区
  __wbg_subarray(0, 32)                      → 取前 32 字节
  __wbg_getRandomValues()                    → 生成 32 字节随机密钥材料
  __wbindgen_memory()                        → 获取 WASM 内存引用
  __wbg_buffer()                             → 获取 ArrayBuffer
  __wbg_new()                                → 新建 Uint8Array 视图
  __wbg_set()                                → 将随机数写入 WASM 内存

  __wbg_instanceof_Window() = true           → 检查 fakeWindow
  __wbg_document()                           → 获取 self.document
  __wbg_querySelector("link[rel*='icon']")   → 第一次读 DOM：查找 favicon
  __wbg_getAttribute("href")                 → 读取 href 属性
      = "https://js-assets.mihuashi.com/.../favicon.ico"

  __wbg_instanceof_Window() = true           → 第二次检查
  __wbg_document()                           → 再次获取 document
  __wbg_querySelector("meta[name='keywords']") → 第二次读 DOM：查找 keywords
  __wbg_getAttribute("content")              → 读取 content 属性
      = "米画师,mihuashi,约稿平台,约稿,插画外包,..."

  __wbindgen_string_new("content")           → 内部使用
  __wbg_get(key="content")                   → 通过 Reflect.get 获取属性值
  __wbindgen_string_get(val="米画师,...")     → 转为 WASM 字符串
=== signtool_new() 返回 ptr=1114520 ===

=== signtool_sign(url="/api/v1/stalls", ts=...) 调用序列 ===
  __wbg_static_PROCESS                       → 检查 process（返回 0 = 不存在）
  __wbindgen_string_new("abort")             → 内部处理
  __wbg_get(key="abort") = undefined
  __wbindgen_is_function = false
  __wbg_subarray(0, 32)                      → 取随机数前 32 字节
  __wbg_getRandomValues()                    → 生成新的随机数
  __wbindgen_memory()
  __wbg_buffer()
  __wbg_new()
  __wbg_set()                                → 写入 WASM 内存
=== signtool_sign() 完成，签名长度 88 ===
```

**关键发现**：`signtool_new()` 读取了 **3 类环境特征**来派生签名密钥：

| 调用序列 | 读取内容 | 作用 |
|---|---|---|
| `getRandomValues()` 两次 | 32 字节随机数 | 随机密钥材料 |
| `querySelector("link[rel*='icon']")` → `getAttribute("href")` | favicon URL | 密钥派生因子 1 |
| `querySelector("meta[name='keywords']")` → `getAttribute("content")` | 页面 keywords | 密钥派生因子 2 |

> **⚠️ 踩坑记录**：最初只补了 favicon link，遗漏了 `meta[name='keywords']`。
> 缺少这个因子的值导致 WASM 密钥派生结果与服务端不匹配，API 一直返回 403 "签名错误"。
> 通过对照上述调用序列逐项比对 stub 实现，定位到缺失项，补上后签名立刻生效。
>
> **核心经验：补环境的第一步不是写代码，而是对目标函数做完整调用追踪，
> 拿到每一步的参数和返回值后，再按图索骥地实现 stub。**

### 5.2 Node.js 环境差异

| 差异 | 浏览器 | Node.js | 处理方式 |
|---|---|---|---|
| `self` / `window` | 存在 | 不存在 | 伪造 fakeWindow |
| `document` | HTMLDocument | 不存在 | 伪造 fakeDocument + fakeFaviconLink + fakeMetaKeywords |
| `crypto.randomFillSync` | 有 | 无 | `require('node:crypto').randomFillSync` 补上 |
| `process` 全局 | 无 | 有 | wbg 中全部返回 0（null），强制走浏览器路径 |

### 5.3 WASM 加载方式

使用同步加载（`new WebAssembly.Module()` + `new WebAssembly.Instance()`），
避免 `WebAssembly.instantiateStreaming` 的异步问题，适配 subprocess 和 execjs 两种调用方式。

## 六、签名实现方案

### 6.1 方案选型

| 方案 | 可行性 | 工作量 | 最终选择 |
|---|---|---|---|
| 纯算（反编译 WASM 用 Python 重现） | 理论可行 | 极大（65KB WASM，170 函数） | ✗ |
| wasmtime-py stub（Python 侧跑 WASM） | 可行但有兼容问题 | 大（42 个 Python stub） | ✗ |
| **Node.js 补环境** | 可行 | 中等（V8 原生兼容 WASM） | ✓ |

### 6.2 补环境 vs 纯算

```
补环境（当前方案）：
  Python → sign_tool.js → 加载原始 WASM → signtool_sign()
           只需要知道 "WASM 要什么输入"
           不需要知道 "WASM 内部怎么算"

纯算（未实现）：
  Python → 自己实现签名算法
           需要知道 favicon+keywords+随机数 → 密钥的派生哈希
           需要知道 signtool_sign 的完整签名算法
           需要逆向 65000 字节 WASM 的全部指令
```

### 6.3 调用流程

```
mhs_crawler.py (配置 CATEGORY, PAGE_COUNT)
  → fetch_categories()          # 获取分类列表（无需签名）
  → mhs_sign.get_request_headers(url_path)
    → sign_url(url_path, timestamp)
      → subprocess.run / execjs.call
        → sign_tool.js
          → 同步加载 WASM
          → signtool_new()      # 初始化密钥（读取 DOM 特征）
          → signtool_sign()     # 对路径和时间戳签名
          → 返回签名字符串
  → requests.get(url, params, headers)
  → 解析 JSON，打印 标题/作者/价格/图片
```

### 6.5 踩坑教训

**补环境的第一步应该是完整调用追踪，而不是直接写 stub。**

在真实浏览器中对 `signtool_new()` / `signtool_sign()` 做完整 trace，
记录每一个 wbg 函数调用的顺序、参数、返回值，然后按图索骥地实现 stub。
少一个函数、参数顺序不对、返回值类型不匹配，都会导致签名失败。
